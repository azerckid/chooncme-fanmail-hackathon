## 3. 강점

- **모듈 분리**: email / llm / scheduler 역할이 나뉘어 있고, index로 재export 되어 의존 관계가 명확함.
- **에러·재시도**: 분류 실패 시 `general`로 폴백, LLM `withRetry`(지수 백오프), 아카이브 실패 시 로그 후 계속 진행.
- **Rate limit 대응**: 분류·답장 생성 간 `delayMs`로 요청 간격 조절.
- **드라이런**: `EMAIL_DRY_RUN=true` 시 실제 발송 없이 로그만 출력.
- **중복 처리**: 아카이브 중복 시 기존 letterId로 답장 생성·예약 가능.
- **이미 답장한 레터**: `sendReplyImmediately`에서 `isReplied` 체크 후 스킵.
- **Graceful shutdown**: start-with-cron에서 SIGINT/SIGTERM 시 스케줄러 정지 후 종료.

## 4. 개선·검토 사항

### 4.1 환경 변수 로드 (권장 수정)

- **현상**: `scripts/start-with-cron.ts`에서 `import 'dotenv/config'`만 사용. 기본값은 **`.env`** 만 로드함.
- **영향**: 설정을 `.env.local`에만 두고 실행하면 DB·Gmail·Gemini 등이 비어 있을 수 있음.
- **권장**: Next/다른 스크립트와 동일하게 `.env.local` 명시.

```ts
// 현재
import 'dotenv/config';

// 권장
import dotenv from 'dotenv';
import path from 'path';
dotenv.config({ path: path.resolve(process.cwd(), '.env.local') });
```

- 로컬에서 `.env`를 쓰는 경우에는 기존대로 두거나, `.env`와 `.env.local` 순서로 로드하도록 선택 가능.

### 4.2 Drizzle update 결과 (확인 권장)

- **위치**: `lib/scheduler/follow-up.ts`의 `cancelFollowUpsForSender`.
- **내용**: `result.rowsAffected`로 취소 건수 사용. Drizzle + SQLite/libSQL에서 `update()` 반환 형태가 드라이버/버전에 따라 다를 수 있음.
- **권장**: 실제 실행 시 취소 로그/카운트가 기대대로 나오는지 확인. 필요 시 Drizzle 문서에 맞게 반환값 처리.

### 4.3 follow_ups 테이블 마이그레이션

- **내용**: `db/schema.ts`에 `followUps` 테이블이 정의되어 있음. DB에 `follow_ups`가 없으면 팔로업 관련 로직에서 오류 발생.
- **권장**: `drizzle-kit generate`로 마이그레이션 생성 후 적용했는지 확인. 적용 전이면 마이그레이션 실행 필요.

### 4.4 Cron API POST 본문

- **위치**: `app/api/cron/check-email/route.ts`.
- **내용**: `await req.json()`로 body 파싱. Content-Type이 application/json이 아니거나 body가 비어 있으면 예외 가능.
- **현재**: try/catch로 빈 객체로 폴백하고 있어 동작은 안정적. GET은 body 없이 상태만 반환하므로 문제 없음.

### 4.5 Gmail 수신 스코프

- **내용**: Gmail API로 메일 **수집**을 하려면 읽기 스코프 필요 (예: `https://www.googleapis.com/auth/gmail.readonly` 또는 `gmail.modify`). OAuth 앱에서 해당 스코프로 재발급한 refresh token을 사용해야 함.
- **권장**: 환경변수만 맞추면 되는지가 아니라, OAuth 동의 화면·스코프에 읽기 권한이 포함되어 있는지 확인.

### 4.6 지연 발송 예약과 PM2 재시작

- **내용**: `scheduleDelayedSend`는 프로세스 내부 `setTimeout` 사용. PM2 재시작 시 예약된 지연 발송은 모두 사라짐.
- **계획서**: “로컬 PM2 단일 프로세스, 재시작 시 해당 배치만 재발송 안 함”으로 이미 기술됨. 동작은 계획과 일치.

## 5. 계획 대비 정합성

- **1시간마다 이메일 확인**: cron `0 * * * *` 및 `processEmails`로 구현됨.
- **팬레터/일반 분류**: LLM 분류 + `partitionEmails`로 구현됨.
- **LLM 답장 생성**: `reply-generator` + 프롬프트로 구현됨.
- **메일 확인 후 10~30분 뒤 답장 발송**: `calculateRandomDelay(10, 30)` 및 `scheduleDelayedSend`로 구현됨.
- **한 곳에서 처리**: Next 서버와 별도로, `scripts/start-with-cron.ts` 하나로 워커 기동 가능. 또는 Next 앱에서 `POST /api/cron/check-email`만 호출하는 방식으로도 동일 파이프라인 실행 가능.

## 6. 권장 조치

1. **즉시 권장**: `scripts/start-with-cron.ts`에서 `.env.local` 로드하도록 수정 (위 4.1).
2. **확인**: `follow_ups` 테이블 마이그레이션 적용 여부 및 `cancelFollowUpsForSender`의 rowsAffected 동작 확인.
3. **운영**: Gmail OAuth 스코프(읽기), `GOOGLE_GEMINI_API_KEY`(또는 OpenAI), `DATABASE_URL` 등 필수 env가 `.env.local`(또는 PM2 ecosystem)에 있는지 점검.
4. **선택**: 에러 알림을 쓰려면 `ALERT_WEBHOOK_URL` 설정. 사용하지 않으면 미설정 시 로그만 남도록 현재 구현으로 충분함.

## 7. 결론

계획한 “1시간 주기 수집 → 분류 → 아카이브 → LLM 답장 생성 → 10~30분 지연 발송”이 코드에 반영되어 있으며, 모듈 분리·에러 처리·드라이런·중복 처리까지 잘 갖춰져 있음.  
위 권장 사항(환경 변수 로드, follow_ups 마이그레이션, Gmail 스코프)만 반영·확인하면 로컬 PM2 환경에서 그대로 운영 가능한 수준으로 판단됨.
