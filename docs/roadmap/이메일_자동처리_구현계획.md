# 이메일 자동 처리 구현 계획서

## 1. 개요

### 1.1 목적

- **배포 없이 로컬 PM2 환경**에서, **단일 애플리케이션**이 다음을 수행하도록 한다.
  1. **1시간마다** 메일함 확인
  2. 수신 메일을 **팬레터 / 일반 메일**로 분류
  3. 팬레터는 아카이브 저장 후 **LLM으로 답장 문구 생성**
  4. **메일 확인 시점 기준 10분~30분 사이**의 랜덤 지연 후 답장 메일 발송

### 1.2 범위

- 실행 환경: 로컬, PM2로 Next.js 앱 상시 구동
- 이메일 수신: Gmail 기준 (Gmail API 또는 IMAP)
- 이메일 발송: 기존 `lib/mail.ts`(nodemailer) 활용
- 아카이브·답장 발송: 기존 `POST /api/letters`, `POST /api/replies/send` 활용
- LLM: 분류·답장 생성용 (예: Google Gemini, OpenAI 등 하나 선택)

### 1.3 비범위

- 배포(원격 서버·Vercel 등) 대상 아님
- 다른 메일 제공자(Outlook 등) 지원은 후순위

## 2. 전체 흐름

```
[매 시 정각 또는 1시간 간격]
    │
    ▼
┌─────────────────────┐
│ 1. 메일함 조회       │  ← Gmail API 또는 IMAP
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 2. 메일별 분류       │  ← LLM: "팬레터" vs "일반"
└──────────┬──────────┘
           │
     ┌─────┴─────┐
     ▼           ▼
  팬레터      일반 메일
     │           └→ (무시 또는 로그만)
     ▼
┌─────────────────────┐
│ 3. 팬레터 아카이브   │  ← POST /api/letters (내부 호출 또는 직접 DB+검증)
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 4. 답장 문구 생성   │  ← LLM (제목·본문 초안)
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ 5. 발송 지연 예약   │  ← 10~30분 사이 랜덤(분) 후 실행
└──────────┬──────────┘
           │
           ▼ (지연 경과 후)
┌─────────────────────┐
│ 6. 답장 메일 발송   │  ← POST /api/replies/send (내부 호출 또는 lib/mail + DB 갱신)
└─────────────────────┘
```

## 3. 스케줄 및 지연 규격

| 항목 | 규격 |
|------|------|
| 실행 주기 | 1시간 간격 (예: 매시 0분, 또는 node-cron `0 * * * *`) |
| 발송 지연 | 메일을 “처리 완료한 시점” 기준 **10분 ~ 30분** 사이 **랜덤** (분 단위 또는 초 단위는 구현 시 결정) |
| 중복 방지 | 이미 아카이브된 `emailId`는 POST /letters에서 409이므로 스킵; 동일 메일에 대해 답장 2회 발송 방지 (replies 테이블·letterId 기준) |
| 중복 시 정책 | 409 응답 시: ① 해당 letterId로 replies 조회 → 답장 없으면 답장 진행, ② 이미 답장 완료면 완전 스킵 |

```
│   ├── email/
│   │   ├── fetch.ts                  # 메일함 조회 (Gmail API 또는 IMAP)
│   │   ├── classify.ts               # 팬레터 vs 일반 (LLM 호출)
│   │   └── types.ts                  # 메일 메타/본문 타입
│   ├── llm/
│   │   ├── client.ts                 # LLM 클라이언트 (Gemini/OpenAI 등)
│   │   ├── classify-prompt.ts       # 분류용 프롬프트
│   │   └── reply-prompt.ts          # 답장 생성용 프롬프트
│   └── scheduler/
│       ├── cron.ts                   # 1시간 주기 스케줄 등록
│       └── delayed-send.ts           # 10~30분 지연 후 발송 큐/타이머
├── scripts/
│   └── start-with-cron.ts            # PM2로 실행할 진입점 (Next + cron 동시 기동)
└── docs/
    └── roadmap/
        └── 이메일_자동처리_구현계획.md  # 본 문서
```

- **진입점**: PM2는 `next start` 대신 `scripts/start-with-cron.ts`(또는 `node scripts/start-with-cron.js`)를 실행하여, Next 서버와 1시간 주기 작업을 한 프로세스에서 기동할 수 있음.
- **대안**: Next 서버와 분리해 `node scripts/run-email-worker.js` 같은 **별도 PM2 프로세스**로 워커만 돌리고, 워커 내부에서 node-cron으로 1시간마다 처리할 수도 있음. 계획서 단계에서는 “한 곳에서 처리”를 전제로 **같은 프로세스에서 Next + cron** 방안을 기본으로 기술.

## 6. 상세 설계

### 6.1 메일 수집 (lib/email/fetch.ts)

- **Gmail API**: `gmail.users.messages.list` + `get` (또는 기존 수집기와 동일한 스코프). 최근 1시간분만 조회하거나, `history`/`after` 기준으로 신규만 가져오기.
- **대안 IMAP**: nodemailer 등으로 INBOX 연결, 최근 N개 또는 특정 시간 이후 메일만 읽기.
- 반환 형식: `{ id, threadId, from, to, subject, bodyPlain, bodyHtml, date, labelIds? }[]` 등 공통 타입 정의 (`lib/email/types.ts`).

#### OAuth2 토큰 갱신 처리

- Gmail API OAuth2 사용 시 **access token 자동 갱신 로직** 필수.
- `googleapis` 클라이언트의 `setCredentials` 호출 시 refresh_token 포함.
- 토큰 만료 감지 → refresh_token으로 새 access_token 발급 → 메모리/환경변수 갱신.
- 장기 운영 시 refresh_token 자체 만료(6개월 미사용 등) 대비 알림 처리 권장.

### 6.2 분류 (lib/email/classify.ts, lib/llm/)

- 입력: 메일 제목·발신자·본문(일부 또는 전체).
- LLM 호출: “이 메일은 팬레터인가, 일반 메일인가?” (예: yes/no 또는 fan/general).
- 출력: `"fan" | "general"` (및 필요 시 신뢰도/코멘트).
- 프롬프트는 `lib/llm/classify-prompt.ts`에 두고, 언어·키워드 기준 가이드 포함.

### 6.3 팬레터 아카이브

- 분류 결과가 "fan"인 메일만 처리.
- `emailId` = Gmail message id (또는 IMAP UID 기반 고유값).
- 기존 `POST /api/letters` 스키마에 맞춰 payload 구성 (senderName, senderEmail, content, subject, receivedAt 등). 필요 시 LLM으로 언어/감정/주제 추출해 채우거나, 일단 필수 필드만 채워서 저장.
- 호출 방식:  
  - **A)** 같은 앱 내에서 `fetch("http://localhost:3000/api/letters", { ... })` + `Authorization: Bearer API_SECRET_KEY`  
  - **B)** 아카이브 로직을 공통 함수로 분리해, API 라우트와 스케줄러가 같은 함수 호출 (권장: 중복 검증·DB 접근 일원화).

### 6.4 답장 문구 생성 (lib/llm/reply-prompt.ts, lib/llm/client)

- 입력: 팬레터 제목·본문·(선택) 발신자 이름.
- LLM 출력: `{ subject: string, body: string }` (plain text). 필요 시 HTML은 본문에서만 사용.
- 톤: 친근·감사 표현 등 정책을 프롬프트에 명시.

### 6.5 발송 지연 10~30분

- “처리 완료 시점” = 아카이브까지 끝난 시점 (또는 답장 문구 생성까지 끝난 시점)으로 정의.
- 해당 시점으로부터 **10분 ~ 30분** 사이 **랜덤** 지연 후 한 번만 발송.
- 구현 옵션:
  - **A)** `setTimeout`으로 지연 후 `POST /api/replies/send` 호출 (또는 공통 발송 함수 호출). 프로세스 재시작 시 예약이 사라지므로, “로컬 PM2 단일 인스턴스” 전제.
  - **B)** “예약 발송 시각”을 DB나 메모리 큐에 넣고, 1분마다 도는 보조 타이머에서 “지금 시각 >= 예약 시각”인 것만 발송. 재시작 시 복구는 DB 저장 시만 가능.

- 계획 단계에서는 **A)** 로 단순화 (PM2 단일 프로세스, 재시작 시 해당 배치만 재발송 안 함).

#### 발송 지연 복구 전략 (권장)

PM2 재시작 시 `setTimeout` 예약이 사라지는 문제 대응:

1. **DB 테이블 추가**: `scheduled_replies`
   ```sql
   CREATE TABLE scheduled_replies (
     id INTEGER PRIMARY KEY,
     letter_id INTEGER NOT NULL REFERENCES fan_letters(id),
     scheduled_at TEXT NOT NULL,       -- ISO8601 발송 예정 시각
     status TEXT DEFAULT 'pending',    -- pending | sent | failed
     subject TEXT NOT NULL,
     body TEXT NOT NULL,
     created_at TEXT DEFAULT CURRENT_TIMESTAMP
   );
   ```

2. **앱 시작 시 복구 로직**:
   - `status = 'pending'` AND `scheduled_at <= NOW()` 인 레코드 조회
   - 즉시 발송 처리 또는 짧은 지연 후 발송
   - 발송 완료 시 `status = 'sent'` 갱신

3. **정상 흐름**:
   - 답장 생성 시 `scheduled_replies` INSERT (status: pending)
   - `setTimeout` 경과 후 발송 → `status = 'sent'` 갱신

### 6.6 답장 발송

- 지연 경과 후 `letterId`, `to`(팬 이메일), `subject`, `body`(LLM 생성)로 기존 발송 경로 사용.
- **A)** `POST /api/replies/send` 내부 호출 (fetch + Bearer)  
- **B)** `lib/mail.sendMail` + DB에 replies insert + fan_letters.isReplied/repliedAt 갱신을 한 함수로 묶어서 재사용 (기존 route와 동일 로직 공유).

## 7. 에러 처리·재시도

| 상황 | 처리 |
|------|------|
| 메일함 연결 실패 | 로그 남기고 다음 주기까지 스킵 (재시도 1~2회 선택) |
| LLM 호출 실패 | 해당 메일만 스킵, 로그; 나머지 메일 계속 |
| POST /letters 409 (중복) | 이미 아카이브된 메일로 간주, 답장만 진행하거나 스킵 (정책 결정) |
| 답장 발송 실패 | 로그; 필요 시 재시도 1회 (지연 후 재발송은 선택) |
| 스케줄러 예외 | try/catch로 한 배치가 전체를 중단하지 않도록, 로그 후 다음 주기 대기 |

## 8. Rate Limit 고려

### 8.1 Gmail API

| 항목 | 제한 |
|------|------|
| 일일 Quota | 10억 units/day (Google Workspace), 개인 계정은 더 제한적 |
| messages.list | 5 units/call |
| messages.get | 5 units/call |
| 권장 | 1시간 주기로 최근 메일만 조회하면 일반적으로 문제없음 |

### 8.2 LLM API

| 항목 | 대응 |
|------|------|
| RPM (분당 요청) | Gemini: 60 RPM (무료), OpenAI: 티어별 상이 |
| TPM (분당 토큰) | 모델·티어별 상이 |
| 대량 수신 시 | 배치 처리 + 요청 간 지연(예: 1초) 삽입 |
| 실패 시 | 지수 백오프(exponential backoff) 재시도 |

### 8.3 대량 팬레터 대응

- 한 주기에 처리할 **최대 메일 수 제한** (예: 50건)
- 초과분은 다음 주기로 이월 (처리 완료 마킹 기준으로 관리)
- 필요 시 **대기열(Queue) 테이블** 도입

## 9. 보안 고려사항

### 9.1 민감 정보 필터링

- LLM에 전달되는 메일 본문에서 **개인정보 마스킹** 검토
  - 전화번호, 주소, 주민번호 등 정규식 기반 필터링
  - 또는 "분류 목적에 필요한 최소 정보만 전달" 정책

### 9.2 발송 메일 스팸 방지

- Gmail 발송 계정의 **SPF/DKIM/DMARC** 설정 확인
- 발송량이 많을 경우 Gmail 일일 발송 제한(500통/일, Workspace는 2000통) 주의
- Reply-To, From 헤더 일관성 유지

### 9.3 API 키 보안

- 환경 변수로만 관리, 코드/로그에 노출 금지
- `.env.local`은 `.gitignore`에 포함 확인

### 11.1 단위 테스트

| 대상 | 테스트 내용 |
|------|-------------|
| `lib/email/classify.ts` | 다양한 메일 샘플로 분류 정확도 검증 |
| `lib/llm/reply-prompt.ts` | 프롬프트 출력 형식 검증 |
| 지연 계산 로직 | 10~30분 범위 내 랜덤 값 생성 확인 |

### 11.2 통합 테스트

- Mock Gmail API 응답으로 전체 파이프라인 검증
- Mock LLM 응답으로 아카이브 → 답장 생성 → 발송 흐름 검증

### 11.3 드라이런(Dry-run) 모드

- `EMAIL_DRY_RUN=true` 설정 시:
  - 메일 수집·분류·아카이브는 정상 수행
  - **실제 메일 발송은 스킵**, 로그만 출력
  - 운영 전 검증 및 디버깅 용도

### 11.4 LLM 분류 정확도 검증

- 테스트 메일 세트(팬레터 20건, 일반 20건) 준비
- 분류 결과와 정답 비교 → 정확도/재현율 측정
- 정확도 90% 미만 시 프롬프트 튜닝
